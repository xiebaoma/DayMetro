# DayMetro 任务排期

## 1. 项目定位

DayMetro 是一个 **2D / 2.5D 日常模拟 + Agent NPC 社交系统**。

项目的核心不是做复杂城市、真实地铁、物理模拟或大型开放世界，而是让玩家在简单场景中和具有状态、记忆、关系、日程和目标的 NPC Agent 互动。

一句话目标：

> 用一个可玩的“一天生活流程”，验证 Agent NPC 是否能让日常记录变得有记忆、有反馈、有陪伴感。

## 2. 开发原则

### Agent 优先

开发时间优先投入到：

- NPC 状态机
- NPC 日程系统
- NPC 记忆系统
- NPC 关系变化
- 玩家与 NPC 的对话交互
- Agent 基于状态和记忆的行为决策
- 每日复盘与长期成长记录

### 场景从简

游戏场景只作为交互容器，不追求复杂表现。

MVP 阶段只需要：

- 宿舍
- 地铁
- 公司
- 食堂
- 操场
- 宿舍夜晚状态

每个场景只保留：

- 可移动区域
- NPC
- 交互点
- 事件触发器
- 简单背景和氛围

不做：

- 真实地铁运行系统
- 城市交通模拟
- 大规模人群行为
- 复杂碰撞和物理
- 高成本美术资源
- 真实工作流程模拟

### 规则控制行为，LLM 负责表达

NPC 的行为边界由规则和状态机决定，LLM 只负责把状态转成自然语言。

例如：

- 是否愿意聊天：由心情、关系、当前任务决定
- 是否触发任务：由日程、记忆、关系决定
- 是否答应邀请：由规则决定
- NPC 说什么：由 LLM 生成

这样可以避免 NPC 被 LLM 带偏，也更容易做成稳定游戏系统。

## 3. 推荐技术路线

### 客户端

- Godot 4
- GDScript
- 2D / 伪 3D / 等距视角
- TileMap 搭建场景
- Area2D 做交互触发
- NavigationAgent2D 做简单 NPC 移动

### 后端

- Python
- FastAPI
- SQLite
- JSON 配置数据
- 后期可接 LangGraph，但不作为第一阶段强依赖

### Agent 与对话

- MVP：规则 Agent + 模板回复
- 原型增强：规则 Agent + LLM 短回复
- 后期：加入记忆检索、轻量 Agent 决策流

### 通信

- MVP 使用 HTTP
- 后期再考虑 WebSocket 推送 NPC 状态

## 4. 最终 Demo 目标

8 周结束时，Demo 应该能完整演示：

1. 玩家从早到晚经历一天流程。
2. 玩家能在宿舍、地铁、公司、食堂、操场与 NPC 互动。
3. NPC 有固定日程，会在不同时间出现在不同地点。
4. NPC 有心情、目标、关系值和记忆。
5. 玩家选择会影响 NPC 对玩家的态度。
6. NPC 会在后续对话中提到过去发生的事。
7. 部分 NPC 会根据状态主动发起提醒、邀请或反馈。
8. 一天结束后生成每日复盘。

## 6. 第 1 周：项目骨架与 Agent 数据设计

### 目标

搭建最小可运行工程，并把 Agent 系统的数据结构先定下来。

这一周不要追求场景好看，重点是把客户端、后端、数据和通信框架搭起来。

### 任务

#### Godot 客户端

- 创建 Godot 4 项目。
- 搭建基础目录结构：
  - `scenes/`
  - `scripts/`
  - `assets/`
  - `ui/`
- 创建主场景 `Main.tscn`。
- 创建玩家节点和基础移动脚本。
- 创建简单 UI 层，用于显示：
  - 当前时间
  - 当前地点
  - 玩家状态
  - 对话框入口

#### Python 后端

- 创建 FastAPI 项目。
- 搭建基础目录结构：
  - `main.py`
  - `npc_agent.py`
  - `scheduler.py`
  - `memory.py`
  - `dialogue.py`
  - `database.py`
- 创建基础接口：
  - `GET /health`
  - `GET /world/state`
  - `POST /dialogue`
  - `POST /event`

#### 数据设计

设计 SQLite 表：

- `npc`
- `npc_memory`
- `player_relation`
- `event_log`
- `save_state`

设计 JSON 配置：

- `npcs.json`
- `locations.json`
- `schedules.json`
- `dialogue_rules.json`

### 交付物

- Godot 项目能启动。
- FastAPI 后端能启动。
- Godot 可以请求后端 `/health`。
- SQLite 数据库能创建基础表。
- 至少有 5 个 NPC 的初始配置。

### 验收标准

- 运行游戏后能看到玩家、时间、地点 UI。
- 后端接口能返回世界状态。
- NPC 的姓名、性格、初始地点、日程、关系值已经可配置。

## 7. 第 2 周：简单场景与基础交互

### 目标

完成轻量游戏壳，让玩家可以在几个核心场景中移动并触发交互。

场景越简单越好，只要能承载 Agent 互动即可。

### 任务

#### 场景搭建

创建 5 个基础场景：

- 宿舍
- 地铁
- 公司
- 食堂
- 操场

每个场景只需要：

- 背景块或简单 TileMap
- 玩家出生点
- NPC 站位点
- 场景出口
- 交互点

#### 玩家交互

- 实现玩家移动。
- 实现靠近 NPC 显示交互提示。
- 实现按键进入对话。
- 实现场景切换。
- 实现基础时间推进。

#### 事件触发

实现基础事件：

- 进入宿舍
- 离开宿舍
- 到达地铁
- 到达公司
- 进入食堂
- 到达操场
- 回到宿舍

事件先写入后端 `event_log`，为后续记忆系统做准备。

### 交付物

- 玩家能从宿舍走到地铁、公司、食堂、操场，再回到宿舍。
- 每个场景至少有 1 个 NPC 占位。
- 玩家靠近 NPC 能打开对话框。
- 关键行为会写入事件日志。

### 验收标准

- 玩家能完成一次简单路线：
  - 宿舍 → 地铁 → 公司 → 食堂 → 公司 → 操场 → 宿舍
- 每次切换地点后，后端世界状态能同步更新。
- `event_log` 中能看到玩家一天的关键事件。

## 8. 第 3 周：NPC 日程与状态系统

### 目标

让 NPC 不再只是站桩角色，而是有自己的时间表、地点、行为、心情和目标。

这一周是 Agent 系统的第一层：规则驱动 NPC。

### 任务

#### NPC 状态模型

每个 NPC 至少包含：

- `id`
- `name`
- `role`
- `personality`
- `current_location`
- `current_action`
- `mood`
- `goal`
- `schedule`
- `relation_with_player`

#### NPC 日程系统

实现固定日程：

- 室友 A：早上宿舍，晚上宿舍打游戏。
- 室友 B：早上宿舍，晚上学习。
- 同事 A：白天公司，中午食堂。
- 同事 B：白天公司，中午食堂，下午吐槽工作。
- 老板 / 导师：上午早会，下午安排任务。

#### 世界时间

实现一天中的关键时间点：

- 07:00 起床
- 08:00 去地铁
- 09:30 到公司
- 10:00 早会
- 12:00 午饭
- 14:00 下午工作
- 18:00 晚饭
- 19:00 返校
- 20:30 操场
- 22:00 洗澡
- 23:00 深夜浏览
- 24:00 睡觉

#### 状态同步

- 后端根据时间更新 NPC 所在地点。
- Godot 根据后端返回显示当前场景中的 NPC。
- NPC 不在当前地点时不显示。

### 交付物

- NPC 会按时间出现在不同场景。
- NPC 当前状态可被后端查询。
- 玩家在不同时间见到的 NPC 不同。

### 验收标准

- 早上宿舍能看到室友。
- 中午食堂能看到同事。
- 下午公司能看到同事和老板 / 导师。
- 晚上宿舍或操场能看到不同 NPC。
- NPC 状态变化会写入事件日志。

## 9. 第 4 周：对话系统与玩家选择

### 目标

实现玩家与 NPC 的核心互动：选项式对话 + 规则控制 + LLM 短回复。

这一周开始体现项目的核心体验。

### 任务

#### 对话入口

- 玩家靠近 NPC 后打开对话框。
- 显示 NPC 名字、头像占位、当前状态。
- 显示 3 到 4 个对话选项。

#### 选项式行为

每次对话提供类似选项：

- 问今天在忙什么
- 邀请一起吃饭
- 吐槽早会
- 关心对方状态
- 结束对话

选项会影响：

- NPC 心情
- NPC 对玩家关系值
- 是否写入记忆
- 是否触发后续事件

#### 规则对话

先实现不依赖 LLM 的规则回复。

例如：

- 关系高：回复更亲近。
- 心情差：回复更短、更冷淡。
- 当前有任务：拒绝长聊。
- 记忆中有相关事件：提到上次发生的事。

#### LLM 短回复

在规则系统稳定后接入 LLM。

LLM 输入包含：

- NPC 身份
- NPC 性格
- 当前地点
- 当前时间
- 当前心情
- 当前目标
- 与玩家关系
- 相关记忆
- 玩家选择或输入

LLM 输出限制：

- 不超过 30 到 50 字。
- 口语化。
- 不越权改变游戏状态。
- 不生成系统未允许的行动结果。

### 交付物

- 玩家能和 5 个 NPC 进行基础对话。
- 对话选项能改变关系值或心情。
- LLM 能根据 NPC 状态生成短回复。
- 没有 LLM 时也能使用规则回复兜底。

### 验收标准

- 同一句玩家选择，在不同 NPC 身上反馈不同。
- 同一个 NPC 在不同心情下回复不同。
- 关系值变化能影响后续对话语气。
- LLM 回复不会直接决定游戏状态，只负责表达。

## 10. 第 5 周：记忆、关系与事件日志

### 目标

实现 DayMetro 最重要的体验之一：NPC 记得玩家做过什么，并在之后的互动中反馈出来。

这一周的重点不是“聊天更聪明”，而是“互动有连续性”。

### 任务

#### 事件日志

记录玩家关键行为：

- 和谁聊天
- 选择了什么选项
- 是否邀请成功
- 是否爽约
- 是否帮助 NPC
- 是否完成任务
- 当天压力和精力变化
- 每日复盘结果

#### 记忆生成

从事件日志中生成 NPC 记忆。

记忆字段包括：

- `npc_id`
- `content`
- `memory_type`
- `importance`
- `created_at`
- `last_used_at`
- `related_event_id`

记忆类型：

- 承诺
- 帮助
- 冲突
- 日常聊天
- 共同经历
- 玩家状态
- 任务进展

#### 关系系统

每个 NPC 对玩家有：

- 关系值
- 信任值
- 冲突值
- 熟悉度

玩家选择会改变这些值。

例如：

- 关心对方：关系 +2
- 帮忙解决问题：信任 +5
- 爽约：信任 -5，冲突 +3
- 连续几天不互动：熟悉度下降

#### 记忆反馈

NPC 对话时会检索相关记忆。

MVP 阶段可以用简单规则：

- 按 NPC 过滤
- 按重要性排序
- 按关键词匹配
- 取最近 3 条记忆

暂时不强依赖向量数据库。

### 交付物

- NPC 能记住玩家至少 3 类行为。
- 第二天对话能提到前一天事件。
- 关系值会持续影响对话。
- 事件日志、记忆和关系可以落库。

### 验收标准

- 玩家昨天帮助同事，今天同事会主动感谢。
- 玩家昨天拒绝室友邀请，今天室友会提到。
- 玩家连续不互动，NPC 态度会变淡。
- 每条重要记忆都能追溯到事件日志。

## 11. 第 6 周：完整一天流程与每日复盘

### 目标

打通完整的一天，让游戏从“系统模块”变成“可玩体验”。

玩家应该能从早上起床玩到晚上睡觉，并看到一天的总结。

### 任务

#### 一天流程

实现主线时间段：

1. 早上宿舍
2. 通勤地铁
3. 公司早会
4. 上午工作
5. 中午食堂
6. 下午工作
7. 晚饭和返校
8. 操场散步
9. 晚上宿舍
10. 睡前复盘

#### 玩家状态

实现基础状态：

- 精力
- 心情
- 压力
- 专注
- 学习值
- 代码值
- 社交值
- 健康值
- 睡眠质量

玩家行为影响状态。

例如：

- 工作：代码值上升，精力下降。
- 午休：精力恢复。
- 聊天：社交值上升。
- 操场散步：压力下降。
- 深夜刷视频：心情上升，睡眠质量下降。

#### 每日复盘

生成每日复盘内容：

- 今日路线
- 重要事件
- 互动过的 NPC
- 关系变化
- 状态变化
- 今日关键词
- 今日总结
- 明天可以继续做什么

每日复盘可以先用规则模板生成，后续再接 LLM 润色。

### 交付物

- 玩家能完整体验一天。
- 一天结束后能看到结算界面。
- 今日事件和状态变化能保存。
- 每日复盘能成为后续 NPC 记忆来源。

### 验收标准

- 一局游戏有明确开始和结束。
- 玩家行为会反映在每日总结中。
- NPC 互动会出现在每日复盘中。
- 第二天可读取前一天的关键事件。

## 12. 第 7 周：轻量 Agent 主动行为

### 目标

让 NPC 不只是被动等待玩家点击，而是能根据日程、关系、记忆和目标主动行动。

这一周是 DayMetro 从“对话系统”升级到“Agent NPC 系统”的关键。

### 任务

#### Agent 决策流

实现轻量决策流程：

1. 感知当前世界状态
2. 读取 NPC 自身状态
3. 检索相关记忆
4. 判断当前目标
5. 选择行为
6. 生成对话或事件
7. 写回状态和记忆

#### 主动行为类型

实现 4 类主动行为：

- 主动打招呼
- 主动提醒
- 主动邀请
- 主动追问过去事件

示例：

- 室友看到玩家晚上回来太晚，主动问是不是又加班。
- 同事记得玩家昨天帮忙修 Bug，今天主动邀请吃饭。
- 老板 / 导师看到任务没推进，主动提醒早会要汇报。
- 车窗里的自己在通勤时询问今天最重要的目标。

#### 行为约束

主动行为必须受规则限制：

- 同一 NPC 不要频繁打扰。
- 同一类记忆不要重复提太多次。
- 当前地点不合理时不能触发。
- 关系不足时不能过度亲密。
- 冲突较高时语气应变化。

#### 可选 LangGraph

如果前 6 周进展顺利，可以尝试把 Agent 决策流抽象成图：

- Perception
- Memory Retrieval
- Emotion Update
- Action Decision
- Dialogue Generation
- State Update

如果时间不足，不接 LangGraph，继续使用 Python 函数式流程即可。

### 交付物

- NPC 能在关键时间主动发起互动。
- 主动互动与记忆、关系、日程有关。
- 玩家能感受到 NPC 在持续观察和反馈自己的生活。

### 验收标准

- 至少 3 个 NPC 有主动行为。
- 至少 1 个主动行为来自过去记忆。
- 至少 1 个主动行为来自当前状态。
- 主动行为不会破坏玩家流程或频繁刷屏。

## 13. 第 8 周：整合、调试与 Demo 打磨

### 目标

把前 7 周的模块整合成一个稳定可演示版本。

这一周不再大幅增加新系统，重点是修 bug、补内容、打磨体验和准备展示。

### 任务

#### 系统整合

- 检查 Godot 与后端通信稳定性。
- 检查存档读写。
- 检查事件日志完整性。
- 检查 NPC 状态更新。
- 检查记忆检索和关系变化。
- 检查每日复盘。

#### 内容补齐

为 5 个核心 NPC 补齐：

- 人设
- 性格
- 日程
- 常用对话
- 记忆触发规则
- 主动行为规则

核心 NPC：

- 室友 A：爱打游戏，外向，容易主动聊天。
- 室友 B：卷王，自律，关注学习和作息。
- 同事 A：友善，适合承接帮助和感谢记忆。
- 同事 B：爱吐槽，适合承接早会、压力和工作情绪。
- 老板 / 导师：任务压力来源，负责目标和反馈。

#### Demo 流程设计

准备一条稳定演示路线：

1. 早上宿舍和室友聊天。
2. 地铁上与“车窗里的自己”对话。
3. 公司早会被老板 / 导师提醒任务。
4. 午饭时和同事聊天并触发关系变化。
5. 下午完成或拖延一个任务。
6. 晚上操场复盘情绪。
7. 睡前生成每日总结。
8. 第二天 NPC 提到前一天发生的事。

#### 体验打磨

- 对话框可读性。
- NPC 气泡提示。
- 时间推进节奏。
- 重要事件提示。
- 每日复盘文案。
- 失败兜底提示。

### 交付物

- 一份可运行 Demo。
- 一条稳定演示流程。
- 5 个可互动 NPC。
- 至少 10 条可被记住的重要事件。
- 至少 5 种关系或记忆反馈。
- 一份每日复盘。

### 验收标准

- Demo 可以连续游玩 15 到 30 分钟。
- 玩家能明显感受到 NPC 有记忆。
- 玩家选择会影响后续对话。
- 场景虽然简单，但不影响核心体验。
- 系统能稳定完成一整天循环。

## 15. 风险与控制

### 风险 1：场景制作耗时过多

控制方式：

- 所有场景先用色块、TileMap 和占位图。
- 不追求美术质量。
- 每个场景只服务一个核心交互。

### 风险 2：LLM 回复不可控

控制方式：

- LLM 不直接改状态。
- 所有状态变化由规则系统决定。
- prompt 限制回复长度和角色边界。
- 保留模板回复兜底。

### 风险 3：记忆系统变复杂

控制方式：

- MVP 不上向量库。
- 先用 SQLite + 重要性 + 时间 + 关键词检索。
- 只记录重要事件，不记录所有废话。

### 风险 4：Agent 行为过度设计

控制方式：

- 先做固定日程。
- 再做状态机。
- 最后做主动行为。
- 不一开始引入复杂多 Agent 框架。

### 风险 5：单人开发范围过大

控制方式：

- 每周只保留一个核心目标。
- 每个模块先做最小可用版本。
- 场景、UI、美术全部让位于 Agent 交互。
- 第 8 周不再加新大功能。

## 17. 核心判断标准

每次开发新功能前都问一句：

> 这个功能是否能增强玩家与 Agent NPC 的互动？

如果答案是否，就先不做。

DayMetro 的核心竞争力不是地图大、场景多、动画复杂，而是：

- NPC 记得你。
- NPC 会因为你之前的选择改变对你的态度。
- NPC 有自己的日程和目标。
- NPC 会在合适的时候主动反馈。
- 玩家能通过一天的互动理解自己的生活节奏。

只要这个体验成立，即使场景很简单，DayMetro 也成立。
